# Design System Strategy: Cyber-Industrial Diagnostic HUD

## 1. Overview & Creative North Star: "The Digital Blueprint"

This design system is not a standard interface; it is a high-precision instrument. We are moving away from the friendly, rounded "SaaS" aesthetic toward a sophisticated, high-fidelity diagnostic environment. Every pixel must feel intentional, as if it were rendered by a real-time telemetry engine.

**Creative North Star: The Digital Blueprint**
The UI should feel like an x-ray of a complex machine. We achieve this through "The Blueprint Effect": using extreme structural precision, razor-sharp edges, and a sense of depth that suggests the interface is projected onto a glass surface rather than printed on a screen. We break the template look by using **intentional asymmetry**—offsetting data readouts and using varied column widths—to mimic technical schematics.

---

## 2. Colors: Tonal Obsidian & Safety Cyan

The palette is anchored in deep blacks to maximize the "glow" of our primary diagnostic color.

-   **Primary (Safety Cyan - #00F0FF):** This is our "Active State" color. Use it sparingly for critical data points, active toggles, and "Laser-Line" accents.
-   **The Neutral Core:** We utilize `surface_container_lowest` (#0E0E0E) for the primary "void" (the base layer) and `surface_container_high` (#2A2A2A) for modular components.
-   **The "No-Line" Rule:** Prohibit the use of 1px solid grey borders for sectioning. To define boundaries, use a background shift (e.g., placing a `surface_container_highest` panel against a `surface` background). If a boundary is mission-critical, use a `primary` glow-line at 0.5px.
-   **The "Glass & Gradient" Rule:** Floating HUD panels must use `surface_variant` with a 60% opacity and a 20px `backdrop-blur`. This creates the "Hardware Lens" effect.
-   **Signature Textures:** Apply a 10% opacity linear gradient from `primary` to `primary_container` on large action surfaces to give them a "powered-on" energy.

---

## 3. Typography: Technical Authority

We pair a high-character geometric sans with a rigid monospace to balance editorial impact with technical legibility.

-   **Display & Headlines (Space Grotesk):** These are your "Structural Headers." Use `display-lg` for top-level diagnostic categories. The wide apertures of Space Grotesk should feel like architectural labeling.
-   **Data & Technical Readouts (JetBrains Mono):** Use this for all numerical values, code snippets, and status logs. It conveys a "Machine-First" logic.
-   **The Hierarchy Strategy:** Headers should always be uppercase with +5% letter spacing to enhance the "blueprint" feel. Body text (`body-md`) should remain clean and highly legible in `on_surface` (#E5E2E1).

---

## 4. Elevation & Depth: Tonal Layering

Traditional shadows have no place in a digital blueprint. We build depth through light and glass.

-   **The Layering Principle:** Depth is achieved by "stacking" surfaces. Place a `surface_container_high` card on top of a `surface_dim` background. The contrast in the obsidian tones provides all the separation required.
-   **The Ghost Border:** For high-density data grids, use a "Ghost Border"—the `outline_variant` token at 15% opacity. It should be felt, not seen.
-   **Ambient Glow:** Instead of dark shadows, "floating" elements (like Modals or Tooltips) should emit a faint `primary` outer glow (`box-shadow: 0 0 15px rgba(0, 240, 255, 0.1)`). This mimics the light bleed of a high-end monitor.
-   **Sharpness:** All `Roundedness` tokens are set to **0px**. Any curvature breaks the industrial immersion.

---

## 5. Components: The Diagnostic Toolkit

### Buttons: High-Visibility Triggers
-   **Primary:** Solid `primary_container` (#00F0FF) with `on_primary` black text. 0px border-radius.
-   **Secondary:** Ghost style. No fill, `primary` 1px border, text in `primary`.
-   **Interaction:** On hover, apply a `primary` outer glow.

### Input Fields: Telemetry Entry
-   **Style:** Underline only (2px `outline_variant`). When focused, the underline transitions to `primary` with a vertical "scan-line" cursor.
-   **Labels:** Always use `label-sm` in `primary` to act as a technical tag.

### Cards & HUD Panels
-   **Constraint:** Forbid divider lines within cards. Use `spacing-6` (1.3rem) of vertical white space to group content.
-   **Background:** Use `surface_container_low` with a subtle 5% opacity "Grid" background-image overlay to reinforce the blueprint theme.

### Status Indicators (Chips)
-   **Visual:** Rectangular blocks with `JetBrains Mono` text.
-   **Logic:** Status "OK" uses `primary`. Status "FAULT" uses `error`. These should look like hardware LEDs.

### New Component: "The Data Ribbon"
A thin, horizontal strip (height: `spacing-1`) that runs across the top of panels, using a `primary` to `primary_container` gradient. Use this to indicate which module of the diagnostic platform is currently "Live."

---

## 6. Do’s and Don’ts

### Do:
-   **Use Asymmetry:** Offset your data columns. Not everything needs to be perfectly centered; technical readouts often feel more authentic when aligned to the left or right margins.
-   **Leverage Monospace:** Use `JetBrains Mono` for any element that changes frequently (clocks, sensor data, percentages) to prevent "layout jitter."
-   **Embrace the Dark:** Ensure `surface_container_lowest` is the dominant color to keep the "Industrial Lab" vibe.

### Don't:
-   **No Rounded Corners:** Never use a border-radius. If a component feels "too sharp," increase the padding; do not round the corners.
-   **No Standard Shadows:** Avoid generic CSS `box-shadow: 0 4px 6px...`. If it needs to lift, use color contrast or a faint neon glow.
-   **No Vibrant Backgrounds:** Avoid using `primary` as a background color for large areas. It is an accent meant to represent light and data, not a structural material.
